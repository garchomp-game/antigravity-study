## Compaction とは

**Compaction（コンパクション）**は、長時間のセッションで膨張したコンテキストを**自動的に圧縮・要約する**仕組みです。

LLMのコンテキストウィンドウは有限です。長い会話を続けると、初期の指示が「遠くなり」、関連性の低い情報が蓄積して**コンテキスト腐敗（Context Rot）**が起きます。

```
セッション開始:
  [System] [Rules] [会話1] [会話2] [会話3]
  → まだ余裕あり

30分後:
  [System] [Rules] [会話1]...[会話20] [ファイル読込] [ツール出力]
  → コンテキストウィンドウが逼迫

Compaction発動:
  [System] [Rules] [要約: 会話1-20の要点] [最新の会話]
  → リフレッシュされて再び効率的に
```

## Compaction の動作メカニズム

<Steps>
1. **閾値検出**
   
   コンテキストのトークン使用量が一定の閾値を超えると、自動的にCompactionがトリガーされる

2. **履歴の要約**
   
   過去の会話履歴を簡潔なサマリに圧縮。重要な決定事項、現在のタスクの状態、ユーザーの要件を保持

3. **計画の更新**
   
   進行中の task.md や implementation_plan.md がある場合、その最新状態を保持
   
4. **コンテキストの刷新**
   
   圧縮されたサマリと最新の会話で、新しいコンテキストを構築。エージェントは「新鮮な状態」で作業を継続
</Steps>

<Aside type="note">
Compactionはユーザーには透過的に行われます。エージェントが突然「忘れた」ように見える場合、Compactionによる情報の取捨選択が原因の可能性があります。重要な情報は Artifacts や Rules に永続化しておくことが対策になります。
</Aside>

## コンテキスト飽和（Context Saturation）

### 問題

コンテキストウィンドウに大量の情報を詰め込むと、モデルの精度が低下します。これを**コンテキスト飽和**と呼びます。

| 原因 | 症状 | 対策 |
|---|---|---|
| 巨大な GEMINI.md | 全会話でトークン消費 | Skillsに分離 |
| 不要なファイルの参照 | ノイズが増加 | 必要なファイルだけ参照 |
| 長い会話の継続 | 初期指示が薄まる | 新しい会話を開始 |
| MCPの大量出力 | トークン枠を圧迫 | 出力のフィルタリング |

### 「Less is More」の原則

コンテキスト管理の最重要原則は**必要最小限の情報を、適切なタイミングで提供する**ことです。

```
❌ アンチパターン: ダンプ型
  「このファイル全部読んで」「このドキュメント全部」
  → コンテキストが無駄に消費される

✅ ベストプラクティス: 外科的精度
  「このファイルの42行目のエラーハンドリングを修正して」
  → 必要最小限の情報で正確な出力
```

## 総合ベストプラクティス

本ガイド全体を通じたベストプラクティスを、7つのレイヤーごとに整理します：

<CardGrid>
  <Card title="① Rules を小さく保つ" icon="pencil">
    GEMINI.md はプロジェクトの「設計契約書」。大きくなりすぎたら Skills に分離。否定制約と具体例を優先。
  </Card>
  <Card title="② Skills を活用する" icon="puzzle">
    反復的に必要な専門知識は SKILL.md に。メタデータ（description）を丁寧に書くことで発見性が向上。
  </Card>
  <Card title="③ KI を自然に育てる" icon="random">
    初日から大量に作ろうとしない。使い続ける中で自然に蓄積。定期的に陳腐化していないかレビュー。
  </Card>
  <Card title="④ Artifacts をレビュードキュメントとして扱う" icon="document">
    Implementation Plan は必ず実行前にレビュー。タスクが複雑なほど、計画段階で修正する方が効率的。
  </Card>
  <Card title="⑤ Workflows で定型作業を標準化" icon="list-format">
    繰り返すタスクは Workflow 化。安全なものには `// turbo` で自動実行。チーム全体で共有可能。
  </Card>
  <Card title="⑥ MCP は必要なときだけ接続" icon="external">
    すべてのツールを常時接続にしない。セッションに必要なツールだけを有効化して、コンテキスト汚染を防ぐ。
  </Card>
  <Card title="⑦ 確定的ツールと併用する" icon="approve-check">
    エージェントの出力を linter、テストスイート、型チェッカーなどの**確定的ツール**で検証。AI + 確定的ツール = 最強の組み合わせ。
  </Card>
</CardGrid>

## コンテキスト管理のメンタルモデル

紹介した7つのレイヤーを統合すると、以下のような**コンテキストのライフサイクル**が見えてきます：

```
┌──────────────── 永続レイヤー ────────────────┐
│                                              │
│  Rules         ← 毎回自動ロード              │
│  Skills        ← 関連時にオンデマンドロード    │
│  Knowledge Items ← 過去の知見を自動参照       │
│  Workflows     ← コマンドで明示的に呼び出し    │
│  MCP Config    ← 設定済みツール接続           │
│                                              │
└───────────────────┬──────────────────────────┘
                    │
                    ▼  会話開始時に合成
┌──────────────── セッションレイヤー ──────────────┐
│                                                │
│  System Prompt + Rules + KI Summaries          │
│  + ユーザーの入力                               │
│  + エージェントの出力・ツール呼び出し            │
│  + Artifacts（計画・タスク・ウォークスルー）     │
│                                                │
│  → コンテキストウィンドウの制限内で管理          │
│  → 必要に応じて Compaction が発動              │
│                                                │
└────────────────────────────────────────────────┘
                    │
                    ▼  会話終了時に保存
┌──────────────── 保存レイヤー ────────────────┐
│                                              │
│  Conversation Log ← brain/<id>/overview.txt  │
│  Artifacts        ← brain/<id>/*.md          │
│  Knowledge Items  ← 自動抽出・更新            │
│                                              │
└──────────────────────────────────────────────┘
```

## よくある質問

### Q: コンテキストが足りなくなったらどうする？

1. **新しい会話を開始する** — 最も確実な方法。Artifactsが前の会話の結果を保持しているので、途切れなく作業を引き継げる
2. **task.md を活用する** — 進捗状態を明示的に記録することで、会話をまたいでも状態を復元できる
3. **重要情報をRules/Skillsに昇格させる** — 何度も繰り返す説明は永続化する

### Q: エージェントが前の会話の内容を覚えていない？

- **Knowledge Items** が自動的に関連知識を提供しますが、すべてを記憶するわけではありません
- 重要な決定事項は GEMINI.md に明記するのが最も確実
- 特定の過去の会話を参照したい場合は、Conversation Logsを通じてアクセス可能

### Q: Skills と Rules の使い分けは？

```
Rules:  常に必要な情報（技術スタック、コーディング規約）
Skills: 特定タスクでのみ必要な情報（デプロイ手順、セキュリティ監査チェックリスト）
```

目安：すべての会話で参照されるなら Rules、特定のタスクでのみ参照されるなら Skills

## これから先の学習

このガイドで学んだ内容を踏まえて、次のステップを推奨します：

1. **自分のプロジェクトに GEMINI.md を作成する** — 最も効果が大きい
2. **定型作業を Workflow 化する** — `/test`、`/lint-fix` から始める
3. **Knowledge Items の蓄積を意識する** — 使い続けることで自然に精度が向上
4. **Artifacts のレビュー習慣をつける** — 特に Implementation Plan は必ず目を通す
5. **Skills を作成してみる** — リポジトリ固有のパターンを SKILL.md に記述

<Aside type="tip" title="最後に">
コンテキスト管理は「エージェントに何を見せるか」を設計する行為です。プロンプトエンジニアリングが「何を聞くか」に焦点を当てるのに対し、コンテキストエンジニアリングは「**何を知っている状態で聞かせるか**」を設計します。これこそが、AIエージェントの出力品質を根本的に左右するスキルです。
</Aside>

## この章のまとめ

- Compaction はコンテキスト膨張を自動的に圧縮する透過的な仕組み
- コンテキスト飽和を避けるには「Less is More」の原則
- 7つのレイヤーを永続/セッション/保存の3層で理解する
- 確定的ツール（linter, test）との併用が最も効果的
- コンテキストエンジニアリング = 「エージェントに何を見せるか」の設計
