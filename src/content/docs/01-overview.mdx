## コンテキストとは何か

AIコーディングアシスタントにおける**コンテキスト（Context）**とは、モデルが一度に「見ている」すべての情報のことです。これには以下が含まれます：

- **System Prompt**: ツールの基本動作を定義する指示
- **ユーザーの指示**: あなたが入力したメッセージ
- **コード**: 開いているファイル、参照されているファイル
- **会話履歴**: 現在のセッションでのやり取り
- **外部ソース**: ドキュメント、Web検索結果、ツール出力

すべてのLLMには**コンテキストウィンドウ**（一度に処理できるトークン数の上限）があり、この有限のリソースをいかに効率的に使うかが、AIの出力品質を決定づけます。

## なぜコンテキスト管理が重要なのか

コンテキスト管理が不十分だと、以下の問題が発生します：

| 問題 | 症状 |
|---|---|
| **コンテキスト飽和** | 無関係な情報が多すぎて、エージェントの応答精度が低下する |
| **コンテキスト不足** | 必要な情報が欠けていて、的外れなコードが生成される |
| **コンテキスト喪失** | セッションをまたぐと、前回の文脈がすべて消える |
| **コンテキスト矛盾** | 古い情報と新しい情報が混在し、一貫性のない出力になる |

## Antigravityの設計思想：Agent-First

Antigravityは「**Agent-First（エージェント第一）**」のIDEです。従来のコーディングアシスタントがインライン補完を中心に据えるのに対し、Antigravityでは**自律型エージェント**が計画・実行・検証を一貫して行います。

この設計思想は、コンテキスト管理に大きな影響を与えます：

- **エージェントは自分で必要な情報を探す** — すべてを手動で渡す必要がない
- **構造化された作業記録（Artifacts）** — 生のチャットログではなく、検証可能なドキュメント
- **Progressive Disclosure（段階的開示）** — 必要なときだけ詳細情報を読み込む
- **永続的記憶（Knowledge Items）** — セッションをまたいだ学習

## コンテキスト管理の7つのレイヤー

Antigravityのコンテキスト管理は、以下の7層で構成されています。各層は異なる役割と寿命を持ちます：

```
┌──────────────────────────────────────────────┐
│  ① System Prompt（不変）                      │
│    → AIの基本動作・ペルソナ・ツール定義         │
├──────────────────────────────────────────────┤
│  ② Rules（永続・ユーザー定義）                 │
│    → GEMINI.md / ユーザー設定ルール            │
├──────────────────────────────────────────────┤
│  ③ Skills（遅延ロード）                       │
│    → SKILL.md + スクリプト / 参照資料           │
├──────────────────────────────────────────────┤
│  ④ Knowledge Items（自動蓄積・永続）           │
│    → 過去の会話から抽出された知識               │
├──────────────────────────────────────────────┤
│  ⑤ Artifacts（会話単位・構造化）               │
│    → 計画書 / タスクリスト / ウォークスルー     │
├──────────────────────────────────────────────┤
│  ⑥ Workflows & MCP（オンデマンド）            │
│    → スラッシュコマンド / 外部ツール接続         │
├──────────────────────────────────────────────┤
│  ⑦ Compaction（自動・透過的）                  │
│    → 長いコンテキストの要約・圧縮               │
└──────────────────────────────────────────────┘
```

### 各レイヤーの特性比較

| レイヤー | 寿命 | 読み込みタイミング | ユーザー制御 |
|---|---|---|---|
| System Prompt | 恒久 | 毎回自動 | なし |
| Rules | プロジェクト寿命 | 毎回自動 | 完全制御 |
| Skills | インストール期間 | 必要時のみ | 完全制御 |
| Knowledge Items | 自動管理 | 関連時のみ | 閲覧・管理可能 |
| Artifacts | 会話単位 | 会話開始時 | 完全制御 |
| Workflows & MCP | 設定期間 | コマンド実行時 | 完全制御 |
| Compaction | 一時的 | 自動 | なし |

## 他ツールとの比較：設計思想の違い

同じ「AIコーディング支援」でも、コンテキスト管理のアプローチは大きく異なります：

| 観点 | Antigravity | Cursor | Claude Code |
|---|---|---|---|
| **パラダイム** | Agent-First IDE | IDE-First + AI | Agent-First CLI |
| **コンテキスト戦略** | 段階的開示 + 永続記憶 | 手動スコーピング (`@`) | セッション + `CLAUDE.md` |
| **記憶の永続化** | Knowledge Items（自動） | なし（手動のみ） | `CLAUDE.md`（手動） |
| **作業の構造化** | Artifacts（計画・タスク・ウォークスルー） | なし | なし |
| **自律性** | 自律的に計画・実行・検証 | ユーザー主導 | 自律的だがCLI限定 |

<Aside type="tip">
この7つのレイヤーを意識して使い分けることで、エージェントの出力品質を大きく向上させることができます。次章以降で各レイヤーを詳しく見ていきましょう。
</Aside>

## この章のまとめ

- コンテキストはAIの出力品質を決定する最も重要なリソース
- Antigravityは7層のコンテキスト管理システムを持つ
- Agent-First設計により、エージェントが自律的にコンテキストを管理する
- 各層には異なる寿命と役割があり、意図的な使い分けが効果的
